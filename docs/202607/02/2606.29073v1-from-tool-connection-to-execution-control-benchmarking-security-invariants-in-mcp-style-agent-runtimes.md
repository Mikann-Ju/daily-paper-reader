## 摘要
模型上下文协议（MCP）风格的生态系统为语言模型应用提供了一种实用的连接层，用于工具、资源、提示和传输。随着智能体从连接转向执行，安全决策通常分散在客户端、服务器、提示、审批对话框、OAuth部署和日志中。本文探讨运行时是否能够在保持类似MCP工作流的同时，使执行层的不变性变得明确且可测试。我们定义了八个不变性：元数据非权威性、基于授权的审批、规范资源、主体绑定、作用域能力调用、源和目标数据流授权、拒绝路径审计以及显式协议状态。我们在HCP中实现了这些不变性，HCP是一种用于MCP风格智能体执行的句柄-能力协议参考运行时，通过主体、资源、授权、能力、句柄、策略决策、数据管道检查和审计条目来表示调用。我们针对两个类似MCP的基线评估了HCP：一个朴素的连接层运行时和一个基于实践信息的连接层缓解基线，后者包含元数据检查、会话检查和每次调用审批。在10个基准测试案例中，朴素基线允许所有模拟攻击，缓解基线允许10个中的6个，而HCP在保留审计证据的同时阻止了所有10个攻击。消融实验确定了哪些运行时组件阻止攻击并保留法医证据。一项本地内存微基准测试报告了测量策略、调用、窥视和管道操作的亚毫秒平均延迟。一个受限的GitHub README筛选样本提供了生态系统信号，而非漏洞发现。结果支持一个狭窄的主张：MCP风格的智能体系统除了连接层约定外，还需要一个执行控制层。

## Abstract
Model Context Protocol (MCP)-style ecosystems give language-model applications a practical connection layer for tools, resources, prompts, and transports. As agents move from connection to execution, security decisions often remain split across clients, servers, prompts, approval dialogs, OAuth deployments, and logs. This paper asks whether a runtime can make execution-layer invariants explicit and testable while preserving MCP-like workflows. We define eight invariants: metadata non-authority, grant-backed approval, canonical resources, principal binding, scoped capability invocation, source-and-target data-flow authorization, deny-path audit, and explicit protocol state. We implement these invariants in HCP, a Handle-Capability Protocol reference runtime for MCP-style agent execution that represents calls through principals, resources, grants, capabilities, handles, policy decisions, data-pipe checks, and audit entries. We evaluate HCP against two MCP-like baselines: a naive connection-layer runtime and a practice-informed connection-layer mitigation baseline with metadata linting, session checks, and per-call approvals. Across 10 benchmark cases, the naive baseline permits all modeled attacks, the mitigation baseline permits 6 of 10, and HCP blocks all 10 while preserving audit evidence. Ablations identify which runtime components block attacks and preserve forensic evidence. A local in-memory microbenchmark reports sub-millisecond mean latencies for measured policy, invocation, peek, and pipe operations. A bounded GitHub README-screening sample provides ecosystem signals, not vulnerability findings. The results support a narrow claim: MCP-style agent systems need an execution-control layer in addition to connection-layer conventions.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：MCP（Model Context Protocol）等连接协议为LLM智能体提供了工具、资源、提示和传输的标准化连接层，但安全决策在客户端、服务器、提示、审批对话框、OAuth部署和日志之间分散，缺乏统一的执行层控制。智能体从连接转向执行时，元数据、审批、资源解析、数据流等关键安全点无法被一个集中的运行时检查，导致多种攻击（如工具投毒、混淆代理、权限滥用、上下文污染、传输状态混淆）容易得逞。
- **整体含义**：论文主张MCP风格的智能体系统除了连接层约定外，还必须增加一个显式、可测试的执行控制层，使安全不变量成为运行时的一等对象，从而可重复测量和审计。

### 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：定义八个执行层安全不变量，并实现一个参考运行时**HCP（Handle-Capability Protocol）**，以中心化的方式中介智能体与提供者之间的执行路径，将授权、资源、能力、句柄、审批、数据流和审计统一建模为运行时对象。
- **八个不变量**（I1~I8）：
  - I1：工具元数据不是策略权威（不能从描述中授予权限）。
  - I2：审批不能绕过授权（高风险操作需要同时具有授权和审批）。
  - I3：调用者提供的资源字符串不能覆盖提供者规范的资源（策略使用提供者解析的资源）。
  - I4：主体绑定到授权、任务、句柄和审计（错误主体不能调用或检查属于其他主体的句柄）。
  - I5：能力调用基于授权（所需作用域触发主体、资源、能力匹配）。
  - I6：数据管道授权源和目标（源句柄所有权和目标能力策略都必须通过）。
  - I7：拒绝路径可审计（失败的调用、窥视、管道和策略决策留下证据）。
  - I8：会话和协议状态显式（未初始化、版本错误、过期会话可预测地失败）。
- **关键技术细节**：
  - HCP运行时包含：Principal（主体）、Resource（提供者规范的资源）、Grant（授权链接主体、作用域、资源和能力）、Capability（能力声明操作、作用域、资源解析行为和审批要求）、PolicyDecision（允许/拒绝判断）、Handle（对输出数据的引用，携带主体、能力、资源、任务元数据）、DataPipe（数据管道，从源句柄到目标能力之间进行所有权和目标策略检查）、AuditEntry（审计条目记录策略决策、句柄访问、管道尝试等）。
  - 调用流程：能力发现→任务开始（主体→能力→输入→选项）→运行时解析规范资源→查找匹配的授权→检查审批（若需要）→提供者执行→输出存入句柄→审计记录。
  - DataPipe流程：源句柄所有权检查→目标能力策略检查（包括数据类兼容性）→允许/拒绝并记录审计。

### 6. 论文的主要结论与发现

- **主要结论**：MCP风格的智能体系统需要**执行控制层**，而不仅仅是连接层约定。HCP参考运行时能够通过集中化的执行不变量阻止所有10个模拟攻击，同时保留完整的审计证据。而朴素基线（B0）允许所有攻击，实践连接层缓解基线（B1）仍允许6个攻击。
- **关键发现**：
  - 连接层缓解（元数据检查、会话检查、每次调用审批）对混淆代理和传输状态类攻击有效，但对工具投毒、过度权限、数据管道攻击无效。
  - 消融实验显示：**授权匹配**是阻止最多攻击（6/10）的关键组件；审批门控、句柄所有权、数据管道目标策略等各自阻断特定攻击。
  - 审计完整性依赖于专用审计组件：禁用审计后，攻击虽被阻止但失去法医证据。
  - 微基准测试显示亚毫秒级延迟（策略解释0.021ms，任务启动0.101ms，句柄窥视0.004ms，数据管道0.118ms），表明执行控制开销对于实际智能体工作流可忽略。
  - 生态系统筛选表明，公共MCP相关GitHub仓库的README中常可见到权威表面暴露和审计故事缺失，凸显了建立可审计执行层的必要性。

### 7. 优点

1. **系统化安全不变量定义**：八个不变量清晰、可测试，覆盖了执行层的主要安全维度。
2. **可重现基准实验**：案例JSON、基准管道和生成脚本公开，支持独立验证。
3. **对比基线设计合理**：区分朴素基线（B0）和实践缓解基线（B1），避免稻草人。
4. **消融实验**：量化各个组件贡献，证明不是单一“魔法开关”导致结果。
5. **审计/可视化**：将审计作为一等对象，强调预防与证据并重。
6. **多个附属实验**：兼容性检查、操作治理表面、生态系统筛选，从不同角度夯实结论。
7. **低开销证据**：微基准测试证实执行控制层对性能影响极小。

### 8. 不足与局限

1. **运行时非生产级**：HCP参考运行时使用内存存储，不具备持久性、生产级沙箱、加密签名、提供者认证等。
2. **攻击模型不完整**：基准测试仅覆盖5类10种攻击，不包括提示注入、恶意模型行为、OAuth客户端可控、供应链攻击、提供者级恶意软件等。
3. **生态系统筛选受限**：仅分析公开README元数据，未执行服务器代码或验证实际漏洞，结论不能推及全部MCP部署。
4. **兼容性有限**：兼容性检查基于自定义MCP类似原语，并非官方MCP认证，不能保证与任意第三方客户端互操作。
5. **无用户研究**：操作治理表面检查仅展示可能性，未进行人机工效学或开发效率的实际测量。
6. **实验规模较小**：基准案例数仅10个，可能遗漏复杂攻击模式；微基准测试环境单一，未测试分布式场景。
7. **隐私/伦理**：生态筛选只使用公开数据，但论文未讨论潜在的数据收集伦理审查（如GitHub仓库内容的中转分析）。
