## 摘要
LLM智能体越来越多地作为基于智能体框架构建的源代码应用程序开发。这些智能体程序将传统宿主语言代码与框架定义的模型、提示、工具、记忆和多智能体编排逻辑语义相结合。因此，它们的行为不仅取决于传统的控制流和数据流，还取决于一类新的智能体依赖。此类依赖通常表现为框架诱导的语义，例如智能体构造函数、工具装饰器和智能体移交声明，这使得现有静态分析或依赖跟踪工具难以恢复。本文提出了AgentFlow，这是第一个用于从智能体程序中恢复和分析智能体依赖的静态分析框架。AgentFlow构建了智能体依赖图（ADG），这是一种框架无关的图表示，将智能体、提示、模型、能力、记忆状态和控制策略表示为类型节点，并将其组件依赖、控制流和数据流依赖捕获为类型边。基于ADG，AgentFlow支持一系列用于智能体治理和安全分析，包括智能体物料清单（BOM）生成和提示到工具风险检测。我们针对五个代表性智能体框架实现了AgentFlow，并在包含5399个真实智能体程序的语料库AgentZoo上进行了评估。评估表明，AgentFlow比现有的基于AST的智能体静态分析工具恢复出更丰富的智能体实体和依赖，生成更具依赖感知的智能体BOM，并在真实智能体程序中发现了238个污点式提示到工具风险。这些结果表明，ADG为理解、治理和保护新兴智能体软件提供了实用基础。

## Abstract
LLM agents are increasingly developed as source-code applications built on agent frameworks. These agent programs combine conventional host-language code with framework-defined semantics for models, prompts, tools, memory, and multi-agent orchestration logic. As a result, their behavior depends not only on traditional control and data flows, but also on a new class of agent dependencies. Such dependencies are often expressed as framework-induced semantics, such as agent constructors, tool decorators, and agent handoff declarations, making them difficult to recover with existing static analysis or dependency tracking tools. In this paper, we present AgentFlow, the first static analysis framework for recovering and analyzing agent dependencies from agent programs. AgentFlow constructs an Agent Dependency Graph (ADG), a framework-agnostic graph representation that represents agents, prompts, models, capabilities, memory states, and control policies as typed nodes, and captures their component-dependency, control-flow, and data-flow dependencies as typed edges. Built on ADGs, AgentFlow supports a range of analyses for agent governance and security, including Agent Bill of Materials (BOM) generation and prompt-to-tool risk detection. We implement AgentFlow for five representative agent frameworks and evaluate it on AgentZoo, a corpus of 5,399 real-world agent programs. Our evaluation shows that AgentFlow recovers richer agent entities and dependencies than existing AST-based agent static analysis tools, generates more dependency-aware Agent BOMs, and uncovers 238 taint-style prompt-to-tool risks in real-world agent programs. These results show that ADG provides a practical foundation for understanding, governing, and securing emerging agent software.

## 1. 核心问题与整体含义（研究动机和背景）
- **问题**：LLM 智能体程序（agent programs）建立在特定框架（如 LangGraph、OpenAI Agents SDK、CrewAI 等）之上，其行为不仅依赖传统控制流和数据流，还依赖一类新型的“智能体依赖”（agent dependencies）。这类依赖由框架诱导的语义（如智能体构造函数、工具装饰器、移交声明）引入，传统静态分析或依赖追踪工具难以恢复和理解。
- **意义**：缺乏对智能体依赖的建模，阻碍了智能体程序的理解、治理和安全保障，例如：
  - 威胁建模：无法清楚哪些工具、记忆、策略可以被访问。
  - 供应链透明：现有 SBOM/AI-BOM 无法捕获智能体特有的组件（如工具、MCP 服务器）及其绑定关系。
  - 安全分析：难以检测提示注入导致的高权限工具调用（prompt-to-tool 风险）。
- **目标**：提出一种系统的方法来建模和静态分析智能体程序中的依赖关系。

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：构建一个框架无关的中间表示——**智能体依赖图（Agent Dependency Graph, ADG）**，将智能体程序中的实体（智能体、提示、模型、能力、记忆状态、控制策略）抽象为类型节点，将组件依赖、控制流依赖和数据流依赖抽象为类型边。
- **ADG 结构**：
  - **Agent Component Dependency Graph (ACDG)**：描述静态组件绑定关系（智能体→模型、提示、工具、记忆等）。
  - **Agent Control Flow Graph (ACFG)**：描述可能的控制转移（如工具调用、智能体移交），可受策略守卫。
  - **Agent Data Flow Graph (ADFG)**：描述信息传播（提示→智能体、智能体→工具、工具返回、智能体间消息、记忆读写等）。
- **构建流程**：
  1. **事实提取前端**：识别框架特定构造（如 `@function_tool`、`HostedMCPTool`、`Agent(...)`、`handoff(...)`），进行别名解析，输出标准化事实（实体定义、组件绑定、控制流、数据流）。
  2. **智能体内部依赖分析**：基于事实构建局部 ACDG/ACFG/ADFG。
  3. **智能体间依赖分析**：处理跨智能体的移交、共享记忆等，形成全局 ADG。
- **应用**：
  - **Agent BOM 生成**：查询 ACDG 获取每个智能体绑定的所有组件。
  - **Prompt-to-Tool 风险检测**：结合 ADFG（提示→智能体数据流）和 ACFG（智能体→工具控制流）识别污点传播路径，前提是 (a,c) 属于数据流边 `E_arg`。

## 6. 主要结论与发现
- AgentFlow 在 ADG-Eval 上成功分析 59/60 项目（1 个超时），恢复的节点数中位数 17（vs AGENT-WIZ 4，AGENTIC RADAR 2.5），边数中位数 28（vs 0, 2）。说明 ADG 能捕获更丰富的实体和依赖。
- AgentFlow 生成的 Agent BOM 在 BOM-Eval 上包含 2295 个组件和 1008 个绑定关系，远超所有基线工具（最多 971 组件、0 绑定关系）。其他 BOM 工具几乎未提供依赖关系。
- 在 AgentZoo 中发现 238 个项目存在 prompt-to-tool 风险（4.4%），共 4357 个风险路径。人工验证精度 73%，漏报 9 个（主要是因自定义命令封装或动态绑定等）。
- 中位分析时间 14.17 秒，95 分位 163.54 秒；中位 ADG 大小 208 节点/206 边，可扩展至数千项目。

## 7. 优点
- **创新性**：首个提出 ADG 作为框架无关的静态分析中间表示，专门捕获智能体程序的新型依赖。
- **实用性**：实现了对 5 个主流框架的解析器，支持大规模真实项目分析，运行时间可接受。
- **全面性**：支持多种依赖类型（组件、控制流、数据流），并基于 ADG 实现了两个有价值的应用（BOM 生成、风险检测）。
- **评估充分**：使用大规模语料库，与多个现有工具定量/定性对比，并有人工验证，结果可信。

## 8. 不足与局限
- **静态分析固有缺陷**：ADG 是过近似（over-approximate），可能将不可执行的路径也纳入依赖，导致误报。
- **覆盖率限制**：
  - 仅支持 Python 和 5 个框架；对其他语言或新框架需要额外前端提取器。
  - 对用户自定义的智能体结构（不遵循标准框架 API）处理能力有限。
- **动态适应不足**：框架 API 频繁更新，语义注册表需持续维护。
- **底层分析缺失**：ADG 不替代传统代码级分析（如函数内语句的污点追踪），对于更细粒度的漏洞检测需结合其他工具。
- **实验未覆盖**：未进行消融实验（如去掉别名解析的影响），误报分析中 25/27 个假阳性源于 sink 语义不精确（如只读文件、只读 SQL），而非 ADG 本身错误，提示需要改进 sink 定义。
