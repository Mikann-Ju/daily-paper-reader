## 摘要
LLM智能体越来越依赖迭代执行，通过规划、工具使用、状态更新和智能体协作来解决问题。虽然这种设计实现了灵活的自动化，但也产生了一类新的故障：当反馈路径未有效约束时，智能体可能反复执行模型调用、工具、工作流转换或智能体交接。我们称此问题为无限智能体循环。无限智能体循环并非普通编程循环；它们源于智能体逻辑、框架语义、运行时观察和终止机制之间的交互。此类故障可将单个请求放大为长时间运行的模型和工具执行，导致成本耗尽、模型拒绝服务、上下文增长以及重复的外部副作用。我们提出IAL-Scan，一种用于检测真实世界LLM智能体项目中无限智能体循环故障的静态分析工具。IAL-Scan将异构智能体代码抽象为与框架无关的智能体中间表示，构建智能体循环依赖图以恢复显式及框架引发的反馈路径，并检查这些路径是否能在无有效边界的情况下反复触及高成本或状态增长的操作。我们在6,549个LLM智能体仓库上评估IAL-Scan。它报告了74个潜在发现，其中人工审查确认了47个项目中的68个无限智能体循环故障，精确率达到91.9%。

## Abstract
LLM agents increasingly rely on iterative execution to solve tasks through planning, tool use, state updates, and agent collaboration. While this design enables flexible automation, it also creates a new class of failures: an agent may repeatedly execute model calls, tools, workflow transitions, or agent handoffs when the feedback path is not effectively bounded. We call this problem Infinite Agentic Loops (IALs). IALs are not ordinary programming loops; they arise from the interaction between agent logic, framework semantics, runtime observations, and termination mechanisms. Such failures can amplify a single request into long running model and tool execution, causing cost exhaustion, model denial of service, context growth, and repeated external side effects. We propose IAL-Scan, a static analysis tool for detecting IAL failures in real-world LLM agent projects. IAL-Scan abstracts heterogeneous agent code into a framework independent Agent IR, builds an Agentic Loop Dependence Graph (ALDG) to recover explicit and framework induced feedback paths, and checks whether these paths can repeatedly reach costly or state growing operations without an effective bound. We evaluate IAL-Scan on 6,549 LLM agent repositories. It reports 74 potential findings, among which manual review confirms 68 IAL failures across 47 projects, achieving 91.9% precision.

- **安全影响**：IALs 可将单个请求放大为长时间运行的模型和工具执行，导致 **API 成本耗尽、模型拒绝服务、上下文窗口增长以及重复的外部副作用**，属于严重的执行安全与可用性问题。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：将异构的 agent 代码（含框架 API、自定义包装器、依赖关系）抽象为框架无关的中间表示（Agent IR），在此基础上构建 **Agentic Loop Dependence Graph (ALDG)**，识别反馈路径，并检查是否有有效边界覆盖该路径。
- **关键技术细节**：
  - **Agent IR 构建**：
    - 从源码和框架行为中提取两类事实：**事实集**（ExecutionUnit、Controller、Invocation、StateUpdate、Bound、ExitRecord）和 **关系集**（owns、calls、updates、guards、exits、constrains、transitions、dispatches、aliases）。
    - 通过 AST 分析、轻量级名称解析、框架行为建模（匹配 API 模式、工具注册、工作流转等）生成事实。
    - 示例：`while finish_reason is None or ...` 变为 Controller (LOOP)，`self.llm._generate()` 变为 LLM_CALL，`messages.append()` 变为 STATE_APPEND。
  - **ALDG 构建**：
    - 节点：12 种（范围、控制器、调用、状态等），从 Agent IR 的保留事实中创建。边：10 种（控制流、调用、工作流转、工具分发、反馈、退出等），从关系集中派生。
    - 对每个控制器节点，注释循环属性（body_cost_kinds、state_growth、exit_kinds、guard_source、bound_sources）。
  - **IAL 检测算法**（Algorithm 1）：
    1. **SCC 发现**：从 ALDG 中提取可能参与重复执行的边（控制流、调用、框架转移、反馈），计算强连通分量，保留非平凡 SCC。
    2. **候选筛选**：仅保留从 agent 入口可达、包含反馈路径且触及高成本或状态增长操作的 SCC。过滤掉有界迭代模式（流消费、分页、测试桩等）。
    3. **控制器与边界验证**：选择控制器并分类（确定性/模型控制/工具控制/外部状态控制等）。检查是否有有效边界（内置上限、框架默认、配置项）覆盖反馈路径。若未覆盖，则视为未受控路径。
    4. **IAL 判定**：路径满足反馈、代价高/状态增长、且无有效边界。补充 LLM 辅助剪枝（可选）以排除非目标循环。

## 6. 论文的主要结论与发现

- **结论 1**：IAL 是真实且严重的 agent 执行安全问题，出现在所有 8 个主流框架中。LangGraph 和 AutoGen 占发现数的 66.2%。
- **结论 2**：失败模式分布：重试无界（25%）、工具调用迭代无界（23.5%）、多智能体对话无界（20.6%）、工作流循环无有效边界（13.2%）、消息重入无界（10.3%）、Runner/委托/评估反馈（7.4%）。
- **结论 3**：所有 68 个失效的共同根因是 **缺少强边界覆盖反馈路径**。工具控制的继续（41.2%）、模型控制的终止（38.2%）、缺少显式退出（33.8%）为主要原因。
- **结论 4**：静态分析优于纯 LLM 方法：IAL-Scan 覆盖全部 68 个 TP，而 coding assistant 仅覆盖 50 个、pure LLM API 仅 23 个，且后者 token 消耗高 4–34 倍。
- **结论 5**：方法各组件均必要：去除框架建模、agent 门控、边界覆盖或良性循环过滤均导致 TP 下降或 FP 增加。

## 7. 优点

- **问题首创性**：首次系统定义 Infinite Agentic Loop，并揭示其为 agent 系统特有的结构化失效。
- **方法先进性**：通过 Agent IR 和 ALDG 实现框架无关的反馈路径恢复，突破现有静态分析无法建模 agent 语义的限制。
- **实用性**：支持 8 个主流框架，实际部署 31 秒/项目的分析速度，91.9% 的精确率，可集成到 CI/CD。
- **实验严谨性**：大规模真实仓库评估（6,549 个），手工审查协议高一致性（94.6%），消融实验、基线对比、稳定性测试完备。
- **可扩展性**：Agent IR 和 ALDG 设计可扩展到更多语言和框架，LLM 辅助剪枝作为可选模块不影响主要分析。

## 8. 不足与局限

- **静态过度近似**：可能产生假阳性。6 个 FP 来自难以静态解析的间接/框架依赖边界（如 `max_steps` 在循环体外、默认 `max_iterations=3`）。
- **语言与框架限制**：仅支持 Python 及 8 个框架。高度自定义的调度器、外部状态停止逻辑、自然语言语义检查无法准确建模。
- **漏检风险**：手动审查确认 7 个 FN，主要涉及 LangGraph 工作流中未识别的边界、手写工具调用循环、多智能体对话缺乏生效边界。
- **LLM 辅助剪枝不稳定**：不同 LLM 或同一模型多次运行结果有波动，故仅作为负过滤辅助，不能完全依赖。
- **依赖框架行为模型**：如果框架版本更新或自定义包装器非常规，模型需更新，否则可能漏检。
