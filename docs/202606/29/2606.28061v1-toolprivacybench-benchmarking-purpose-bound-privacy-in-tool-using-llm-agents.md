## 摘要
大语言模型（LLM）已从独立的文本生成系统发展为调用外部工具、访问环境、执行多步骤任务的智能体。然而，传统的函数调用基准主要评估任务完成度和API正确性，而隐私评估基准通常关注最终响应或隐私判断。这两种视角都没有捕获跨多个工具执行轨迹中的目的绑定信息流。受当前智能体评估中这一局限性的启发，ToolPrivacyBench审查任务私密原子是否仅被路由到授权的工具和下游接收者，从而评估工具使用过程中的任务完成情况和隐私过度披露。该基准包含2150个案例，其中1150个是完全合成的隐私敏感业务工作流，1000个来自现有的多工具和函数调用基准。每个案例由一个策略知识库表示。智能体针对模拟的业务后端执行后，评估器将记录的工具参数和后端审计日志与该策略知识库进行比较。评估覆盖了九个广泛使用的智能体，以表征目的绑定隐私过度披露。结果表明，成功的工具执行并不意味着适当的隐私披露：智能体可能在完成任务的同时通过中间工具调用传输不必要的私密信息。因此，ToolPrivacyBench形式化了一种“按需知晓”的披露边界，在此边界下，每个工具应仅接收其既定目的所需的信息，并使用轨迹级审计来识别多工具工作流中的隐私过度披露。

## Abstract
Large language models (LLMs) have increasingly moved from standalone text generation systems to agents that invoke external tools, access environments, and execute multi-step tasks. However, conventional function-calling benchmarks mainly evaluate task completion and API correctness, while privacy evaluation benchmarks typically focus on final responses or privacy judgments. Neither perspective captures purpose-bound information flow across an executed multi-tool trajectory. Motivated by this limitation in current agent evaluation, ToolPrivacyBench audits whether task-private atoms are routed only to authorized tools and downstream sinks, thereby evaluating both task completion and privacy over-disclosure during tool use. The benchmark contains 2,150 cases, including 1,150 fully synthetic privacy-sensitive business workflows and 1,000 cases adapted from existing multi-tool and function-calling benchmarks. Each case is represented by a policy knowledge base. After an agent executes against mock business backends, the evaluator compares recorded tool arguments and backend audit logs with this policy knowledge base. The evaluation covers nine widely used agents to characterize purpose-bound privacy over-disclosure. The results show that successful tool execution does not imply appropriate privacy disclosure: an agent may complete a task while transmitting unnecessary private information through intermediate tool calls. ToolPrivacyBench therefore formalizes a need-to-know disclosure boundary, under which each tool should receive only the information necessary for its stated purpose, and uses trajectory-level auditing to identify privacy over-disclosure in multi-tool workflows.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：现有的LLM智能体（Agent）基准主要评估任务完成度和API调用正确性，而隐私评估基准只关注最终响应中的隐私泄露或模型级别的隐私判断。这两种视角都没有捕获**多工具工作流执行轨迹中**的**目的绑定（purpose-bound）信息流**。具体地，同一个私密字段（如症状、身份证号）对于一个工具是必需的，但对于另一个工具可能是多余的甚至禁止的。因此需要评估智能体在调用工具时是否遵守“按需知晓（need-to-know）”的披露边界。
- **背景**：LLM智能体越来越多地用于医疗、金融、保险、税务、招聘、IT运维等业务流程，这些场景涉及多种私密信息和不同接收者。现有的Agent安全基准重点关注恶意提示注入、不安全行为等对抗性威胁，而非**非对抗性的隐私过度披露**——即用户在良性任务中，智能体在不必要的情境下发送了私密信息。
- **整体含义**：工具调用成功 ≠ 适当的隐私披露。评估智能体不仅要看任务是否完成，还要看工具调用轨迹中私密信息是否只流向被授权的工具和接收者（sink）。ToolPrivacyBench正是为了填补这一评估空白而设计。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式

- **核心思想**：将隐私合规问题建模为**基于知识的审计**问题。为每个用例构建一个**策略知识库**，记录当前任务的私密原子（private atoms）、工具目的、信息接收者（sink）类型、以及允许/禁止的字段-工具关系。智能体执行任务后，评估器将工具实际接收的参数和后端审计日志与策略知识库进行对比，判断每个私密原子是否被路由到了未授权的工具或接收者。
- **关键技术细节**：
  - **形式化定义**：多工具任务表示为 `W = (x, T, P, A)`，其中 x是用户请求，T是工具集，P是当前任务的私密原子集，A是字段-工具授权矩阵（0/1矩阵，A[j,i]=1表示私密原子pj允许被工具ti接收）。
  - **工具表示**：每个工具 ti = (name_i, purpose_i, schema_i, sink_i)，sink_i 可以是后端记录、支付系统、内部工单、通知、协作笔记、团队交接等。
  - **轨迹审计**：智能体执行产生轨迹 `τ = [(t1,a1), ..., (tK,aK)]`，披露检测器 D(pj, ak) 判断私密原子是否出现在参数ak中，然后计算过度披露 `O[j,k] = D(pj, ak) * I[A[j,i(k)]=0]`。
  - **评估管道**：Policy-KB Construction → Trajectory Capture → Disclosure Detector → Authorization Reasoner → Metric Aggregator。
- **算法流程**（文字描述）：对每个智能体和每个用例，加载任务、工具、私密原子、授权矩阵；在OpenClaw堆栈上执行智能体；记录工具调用轨迹和后端日志；计算效用指标（TaskSuccess）和隐私指标（FOR, SWLR, ToolFOR, FreeTextFOR, MidFOR, MT-POI, SMTC）。
- **核心指标**：
  - **TaskSuccess**：任务完成度（几何平均 of 任务结果、工作流覆盖、必要事实传递）。
  - **FOR**（Field Opportunity-Normalized Over-Disclosure Rate）：在执行的工具调用中，禁止的字段-工具机会里有实际披露的比例。
  - **SWLR**（Severity-Weighted Leakage Rate）：带严重性加权的FOR。
  - **ToolFOR**：每个工具层面的FOR。
  - **FreeTextFOR**：自由文本字段（如message, note, summary）中的过度披露率。
  - **MidFOR**：中间工作流步骤的过度披露率。
  - **MT-POI**：综合隐私过度披露指数（五个分量加权和）。
  - **SMTC**（Safety-Maintained Task Completion）：TaskSuccess * (1 - MT-POI/100)，综合效用与隐私的分数。

## 6. 论文的主要结论与发现

- **发现1**：当前LLM智能体能够以较高成功率完成多工具任务（TaskSuccess在92-98%之间），但**任务完成与隐私合规是两个独立的能力**。例如，Qwen3.6-plus、Kimi K2.5等模型任务成功率高达97.7%，但MT-POI高达27-28，远高于Gemini 3.5 Flash的19.19。
- **发现2**：隐私过度披露主要集中在**协作类接收者（sink）**——工单（ticket）、交接（handoff）、评论/笔记（comment/note），而不是主要的后端记录。工单的FOR高达51.43%，交接为34.79%。
- **发现3**：自由文本字段（message, description, note, summary, work_notes）是重要的泄露渠道。FTSlotRate超过80%，description的FreeTextFOR达45.82%。
- **发现4**：工作流路径分析表明，记录→工单、文档→通知等转换是最常见的首次泄露点，风险转移率超过82%。
- **发现5**：不同私密原子类型和领域有显著差异。安全敏感凭据、医疗-财务信息、身份证号等在高风险接收者中过度披露更严重。
- **统一结论**：API正确性 ≠ 隐私合规；需要轨迹级审计来识别多工具工作流中的目的绑定隐私过度披露。

## 8. 不足与局限

- **实验覆盖**：仅评估了9个智能体，且为闭源或开源的特定版本，未涵盖更广泛的模型（如Llama、Mistral等）。
- **部署环境假设**：假定模拟后端可信、日志不被篡改，且用户请求为良性（非对抗性）。实际生产环境可能面临更复杂的威胁（如对抗性提示、工具被篡改、审计日志不可信）。
- **检测粒度**：自由文本中的语义变体和隐含引用可能未被完全捕获（如同义词、指代），可能导致漏报。
- **策略知识库的构建**：当前依赖人工标注和规则检查，在扩展到新领域或新工具时成本较高。
- **未考虑长期记忆或跨任务泄露**：仅聚焦于当前任务的原子泄露，不涵盖智能体记忆中的历史信息泄露。
- **缺乏重复性实验**：没有报告多次运行的标准差或置信区间，实验的统计稳健性可进一步提升。
- **资源消耗未报告**：缺少计算资源细节，限制了可复现性评估。
