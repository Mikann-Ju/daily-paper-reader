## 摘要
大型语言模型编码工具赋予代理广泛的文件和shell访问权限，但指导它们的配置层——规则文件、代理定义、集成开发环境特定的标记——在很大程度上缺乏管理。一项对10,008个公共GitHub仓库（n=6,145个代理配置文件）的流行程度研究发现，代理配置作为未声明的共享组件传播：10.1%的追踪路径在不同独立仓库中是SHA-256精确重复（经fork调整，阈值无关），75.5%的克隆对跨越组织边界。另外两种模式具有指示性：配置很少被修改（58%为单次提交；年龄标准化的每月提交次数为0.4 vs CI/CD工作流的0.6），并且很少声明权限边界（代理配置中<1% vs Actions工作流中33%，n=31个真正例）。我们提出一个在工具之上的确定性控制平面，该平面与这些差距一一对应。Rel(AI)Build将代理定义视为受管理的供应链（SHA-256内容寻址、HMAC标记的锁文件、哈希链审计日志）；在调用大型语言模型之前强制实施层级化权限和攻击衍生的阻止列表；通过具有需求到文件到测试可追溯性的阶段状态机对功能工作进行门控；将单一规范定义编译到七个集成开发环境目标；并通过Jaccard相似性检测提示漂移。对注入违规的一致性测试证实每个机制都强制执行其陈述的不变性；开发人员成果留作未来工作。该层的治理必须是确定性的且与工具无关——而不是委托给进一步的大型语言模型编排。

## Abstract
LLM coding harnesses grant agents broad file and shell access, yet the configuration layer that steers them -- rules files, agent definitions, IDE-specific markdown -- is largely unmanaged. A prevalence study of 10,008 public GitHub repositories (n=6,145 agent config files) finds that agent configurations propagate as undeclared shared components: 10.1% of tracked paths are SHA-256 exact duplicates across independent repositories (fork-adjusted, threshold-independent), with 75.5% of clone pairs crossing organisational boundaries. Two further patterns are indicative: configurations are rarely revised (58% single-commit; 0.4 vs 0.6 commits/month age-normalised against CI/CD workflows), and rarely declare permission boundaries (<1% of agent configs vs 33% of Actions workflows, n=31 true positives).   We propose a deterministic control plane above the harness that maps one-to-one to these gaps. Rel(AI)Build treats agent definitions as a managed supply chain (SHA-256 content addressing, HMAC-stamped lockfiles, hash-chained audit logs); enforces tiered permissions and attack-derived blocklists before LLM invocation; gates feature work through a phase state machine with requirement-to-file-to-test traceability; compiles a single canonical definition to seven IDE targets; and detects prompt drift via Jaccard similarity. Conformance tests on injected violations confirm each mechanism enforces its stated invariant; developer outcomes remain future work. Governance of this layer must be deterministic and tool-agnostic -- not delegated to further LLM orchestration.

# 论文总结：《LLM编码代理的确定性控制平面》

## 1. 论文的核心问题与整体含义

- **研究动机**：当前LLM编码工具（如Cursor、Claude Code、GitHub Copilot等）赋予代理广泛的文件与shell访问权限，但指导代理行为的配置层（规则文件、代理定义、IDE特定markdown）却缺乏系统管理。该配置层是“薄治理表面”，存在三个结构性缺口：
  1. 配置未经管理：规则文件被随意复制、编辑、信任，无供应链接保证（无版本、无完整性校验）。
  2. 执行无边界：没有强制过程（无需求→实现→测试追踪，无审查点，自纠循环无硬上限）。
  3. 专业知识不移植：不同IDE有各自配置方言，组织使用多种工具时必须重复编写和维护相同知识。
- **整体含义**：需要将LLM编码代理的配置与过程层视为受管理的软件供应链，用确定性（非LLM）的治理机制来弥补上述缺口，从而实现完整性、可追溯性和可移植性，且独立于底层工具或模型。

## 2. 论文提出的方法论

- **核心思想**：提出一个位于现有LLM工具之上的确定性控制平面（参考实现：Rel(AI)Build），不替换工具，而是增加治理层。所有治理机制均用普通可测试代码实现（哈希、正则匹配、状态机、集合运算），不委托给LLM。
- **关键技术细节**：
  1. **供应链接架构（§4.1）**：
     - SHA-256内容寻址：每个资源有哈希值，安装时重新计算并校验，防止上下文投毒。
     - HMAC-SHA256锁文件：记录已安装资源，写入时用机器本地密钥生成HMAC，检测意外损坏（非对抗性签名）。
     - 追加式哈希链审计日志：每条记录包含自身SHA-256哈希，可验证篡改。
  2. **确定性权限模型（§4.2）**：
     - 五级权限层级（readonly、scribe、operations、specialist、orchestrator），每级有严格工具允许列表。
     - 写入路径验证：规范化路径、拒绝`..`遍历、检查前缀。
     - 代理级硬约束（如git-ops代理禁止修改`.git/config`或`.git/hooks/`）。
  3. **攻击衍生命令/路径阻止列表（§4.3）**：
     - 基于已知供应链攻击（如npx自动确认、curl | bash、持久化路径等）的正则模式。
     - 部署为IDE运行时钩子（Claude Code PreToolUse、Cursor beforeShellExecution），扫描shell命令和写入目标。
  4. **分阶段门控生命周期（§4.5）**：
     - 八个连续阶段（0-Spec,1-Design,3-Implement,4-Test,5-Review,7-Security,8-Finalize）,包含强制HITL门。
     - 状态机强制顺序不变性、委托收据、安全扫描门、3次自纠循环上限。
     - 需求→文件→测试可追溯性：通过`trace.json`记录每个验收准则的实现文件与测试，超出范围的文件触发范围蔓延警告（合作依赖）。
  5. **令牌标准化与漂移检测（§4.4）**：
     - 标准化：CRLF→LF, 去尾空格, tab→空格, 压缩空白行。

  6. **一次定义、编译到多目标（§4.6）**：
     - 单一规范定义经标准化后编译到7种IDE格式（Cursor、Claude Code、VS Code/Copilot、Codex、Windsurf、OpenCode、Kiro）。

## 7. 优点

- **清晰的问题识别与证据**：通过大规模公开仓库实证测量了治理缺口，特别是“配置作为未声明共享组件”的稳健发现（SHA-256阈值独立）是强有力的贡献。
- **创新的治理框架**：将软件供应链安全原则（SHA-256、HMAC锁文件、审计日志）系统应用于代理定义层，这是新颖的。
- **确定性设计原则**：所有治理机制均用确定性代码实现（非LLM决策），使其可测试、可审计、不可绕过。硬编码的门控逻辑（状态机、权限验证）是核心优势。
- **平衡的范围界定**：明确承认机制不解决提示注入、不验证语义正确性、不保护模型提供者基础设施等，诚实披露局限性。
- **可操作实现**：提供了参考实现Rel(AI)Build，包括CLI和MCP服务器，以及明确的阶段门控生命周期、一次定义多目标编译等具体流程。

- **实验局限性**：
  - **缺乏开发者成果评估**：未进行受控实验比较裸工具与受治理工具在生产力、缺陷率、认知负荷等方面的差异。附录D的N=1演示不能作为有效性证据。
  - **漂移阈值未经验证**：Jaccard阈值（0.60/0.80）是操作默认值，未通过行为实验校准（例如SWE-bench相关验证）。

  - **一致性测试范围窄**：仅测试机制是否拦截注入违规，未测试误报率、性能开销、与不同工具链的集成稳定性等。
- **架构局限性**：
  - **提示注入未解决**：高度巧妙的受合规范围攻击可能绕过自动检查，依赖人类HITL门作为主要后盾。
  - **合作追溯边界**：需求→文件→测试追溯依赖代理自愿调用trace-update，恶意代理可绕过。控制平面仅能在后续检查中发现，不能预防。
  - **本地完整性非对抗性**：默认HMAC锁文件使用本地机器密钥，同一用户进程可读取并伪造签名，不抵抗特权攻击者。
  - **工作空间信任边界**：不受信任的第三方仓库（含恶意配置）仍能影响模型行为，运行时钩子和git约束降低但不能消除风险。
  - **阻止列表需主动维护**：基于已知攻击模式，对新型攻击无效。
  - **目标IDE执行不一致**：编译保证语义正确输出，但不同IDE对权限的执行力差异大（Claude Code强，Codex/OpenCode弱），组织需自行审计。
- **可泛化性**：论文基于单一参考实现（Node.js），且流行度研究主要集中在Claude配置（5,441个文件）和Cursor规则，其他工具的代表性有限。声称一次定义编译到多目标，但仅测试7种目标，部分目标（如Kiro）为自定义，实际可用性未证实。
