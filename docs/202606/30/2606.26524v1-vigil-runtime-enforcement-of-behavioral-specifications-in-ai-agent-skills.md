## 摘要
代理系统越来越多地通过第三方技能行动，允许模型生成的决策影响文件、通信渠道和网络物理设备。这些技能通常包含自然语言规范，定义访问权限、披露限制、执行权限和所需的前提条件。尽管此类规范描述了技能行为的预期边界，但它们本身并不提供可执行的运行时执行。执行这些规范带来了上下文粒度挑战：即使为特定任务上下文编写了策略，监控器仍必须决定观察哪些事件、保留什么状态、在执行的哪个范围内推理以及在哪里干预。选择错误的粒度可能会阻塞良性执行或遗漏仅跨多个动作出现的违规。然而，大多数现有的执行机制都假设一个固定的事件模型或执行点。

在这项工作中，我们提出了VIGIL，一个端到端的代理系统运行时执行框架。VIGIL根据来自技能规范、操作员定义的约束以及跨多个技能的全局规则的行为策略，检查代理的实际执行轨迹。为了使此类策略可执行，VIGIL引入了一种策略语言，该语言捕获代理-工具事件上的上下文特定执行要求，包括时间依赖、参数约束和值流条件。该语言与符号评估规则配对，将策略转化为有限轨迹上的SMT约束，使VIGIL能够检测依赖于事件顺序、参数关系或跨调用值流的违规，而不是依赖固定的单调用过滤器。在涵盖办公文档、运营和工程任务的实际LLM代理运行中，VIGIL检测策略违规的召回率超过95%，假阳性率低于10%。

## Abstract
Agentic systems increasingly act through third-party skills, allowing model-generated decisions to affect files, communication channels, and cyber-physical devices. These skills often include natural-language specifications that define access permissions, disclosure limits, execution privileges, and required preconditions. Although such specifications describe the intended boundaries of skill behavior, they do not by themselves provide executable runtime enforcement. Enforcing them raises a contextual granularity challenge: even when a policy is written for a particular task context, a monitor must still decide which events to observe, what state to retain, how far across the execution to reason, and where to intervene. Choosing the wrong granularity can either block benign executions or miss violations that emerge only across multiple actions. Most existing enforcement mechanisms, however, assume a fixed event model or enforcement point.   In this work, we present VIGIL, an end-to-end runtime enforcement framework for agentic systems. VIGIL checks an agent's actual execution trace against behavioral policies from skill specifications, operator-defined constraints, and global rules spanning multiple skills. To make such policies executable, VIGIL introduces a policy language that captures context-specific enforcement requirements over agent-tool events, including temporal dependencies, argument constraints, and value-flow conditions. The language is paired with symbolic evaluation rules that translate policies into SMT constraints over finite traces, allowing VIGIL to detect violations that depend on event order, argument relationships, or cross-call value flow rather than relying on fixed single-call filters. On real LLM-agent runs spanning office-document, operational, and engineering tasks, VIGIL detects policy violations with over 95% recall and a false-positive rate below 10%.

### 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：**分离行为规范的一般含义与其运行实例化**。将被监控的执行轨迹抽象为带类型的事件序列，从该轨迹中提取“签名”（action、参数名、值等），然后将自然语言规范编译为使用该签名的策略语句，最后通过 SMT 求解检查策略是否被违反，并利用不满核心定位违规事件。
- **关键技术细节**：
  1. **轨迹抽象（ABSTRACT）**：将原始工具调用转换成带类型的事件 `e = (a(e), e[·], s(e), o(e))`，其中 `a(e)` 是动作名，`e[·]` 是参数映射，`s(e)` 是成功/失败状态，`o(e)` 是受限输出观察。事件按执行顺序形成轨迹 `τ = e1 ≺ e2 ≺ ...`。轨迹中所有出现的动作、参数名、值构成签名 Σ = (A, K, V)。事实库 JτK 包含所有原子事实（如 `a(e)=warpAffine`、`e3[transform]=M`、`e3 ≺ e5`）。
  2. **策略语言**：包含模式（pattern）和时序形式（temporal forms）。模式用 `a∈α`、`k∈S`、`k=x`（变量绑定）、`o∼p` 等选择事件。时序形式包括：
     - `abs(ψ)`：禁止模式 ψ 出现。
     - `prec(ψ, ψ')`：ψ 必须在 ψ' 之后出现。
     - `resp(ψ, ψ')`：ψ 必须后面有 ψ' 响应。
     - `brespℓ(ψ, ψ')`：响应限于 ℓ 个事件内。
     - `rslv(ψ, ψ')`：最终状态必须解决。
     - `until(ψ, ψ', ψb)`：在 trigger ψ 后，直到 resolving ψ' 前禁止 bad ψb。
     - 回退形式 `□Θ` 允许任意时序体（含 ♢⁻、♢⁺）。
  3. **符号编译**：策略 φ 被编译为 SMT 公式 Rφ，对所有事件求值。例如 `prec(ψ, ψ')` 生成为 `∀e (ψ(e) ⇒ ∃e'≺e ψ'(e'))`，展开成有限事件集上的无量词公式。
  4. **SMT 检查**：使用 Z3 求解器检查事实库 JτK 与 Rφ 是否不可满足。若不可满足，则违反；不满核心直接定位违规事件。
  5. **运行流程**（Algorithm 1）：
     - 输入：执行 L、规范 D。
     - 输出：safe 或 unsafe + 违规事件集 W。
     - 步骤：(Σ, JτK) ← ABSTRACT(L)；P ← COMPILE(D, Σ)；对每个 φ∈P，编码 Rφ，若 unsat(JτK ∧ Rφ) 则加入 W 并定位 loc(φ)；返回 verdict。

### 7. 优点：方法或实验设计上的亮点

- **形式化基础强**：策略语言基于有限时序逻辑，编译为 SMT，保证每个判断的确定性，并可通过不满核心精确定位违规事件。

- **实验全面**：包含人工标注的宏观基准、跨基准泛化测试、真实世界部署研究、消融实验和错误分析，验证了方法有效性及边界。

### 8. 不足与局限：包括实验覆盖、偏差风险、应用限制等

- **观测边界**：VIGIL 只能看到作为工具调用的事件，策略相关证据可能藏在内部脚本体中（如 `secure_archive.py` 内操作不产生事件），导致漏报。
- **编译保真度**：LLM 编译自然语言规范时可能过度泛化（例如丢失“批量”修饰词），导致对良性操作误报。
- **依赖完整轨迹**：必须在执行完全记录后才能判断义务型违规（如 `resp`、`until`），在实时拦截场景中可能需要缓存 pending 调用直到可判断。
- **扩展性**：SMT 查询的规模随事件数线性增长，但文中仅测试了平均几十个事件的轨迹（最大约 60 事件），大规模长轨迹下效率需进一步验证。
- **实验覆盖面**：
  - 仅使用了三个基准（SB+SI、AgentDojo、SafeAgentBench），后者本身偏向单动作违规，不能充分体现 VIGIL 的优势。真实世界研究仅 216 次执行，且局限于部分厂商的技能包。
  - 未进行跨不同 LLM 代理（如 GPT-4、Gemini）的重复实验，可能存在代理偏差。
  - 未测试对抗性攻击下的鲁棒性（如攻击者故意构造使 VIGIL 违背策略的输入）。
- **部署依赖**：需要代理运行环境提供可信执行流记录和拦截能力，不是所有代理框架都原生支持。
