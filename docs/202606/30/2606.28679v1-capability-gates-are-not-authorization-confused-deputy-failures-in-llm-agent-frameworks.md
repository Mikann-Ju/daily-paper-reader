## 摘要
使用工具的LLM代理在持有支付、电子邮件、CRM和基础设施API等副作用工具时，越来越多地读取不受信任的内容，然而常见的框架默认设置仍然将工具暴露与授权混为一谈。我们审计了LangChain/LangGraph、LlamaIndex和Stripe Agent Toolkit是否在执行前，针对每个模型发出的调用（带上具体的参数值）进行重新授权。在固定的公共源代码提交版本中，三者都默认提供了能力门控，但默认情况下没有提供确定性的闭失败单次调用值授权门控。我们引入了ScopeGate，一个用于代理工具调用的五阶段PDP/PEP：范围、授权、金额上限、幂等性和默认拒绝。评估显示，相同的未经授权的支付调用在LangChain的默认派发下执行（附带一个LlamaIndex PoC），但被ScopeGate拒绝；测试的控制组报告0/48静态绕过、0/29未经授权的尝试（40次迭代自适应运行）、0/10良性误拒绝，以及Latam-GPT支付代理在10/10的情况下被遏制。ASR表示尝试的未授权动作，遏制并非治愈，部署层级的声明是基于测量模型类的推理，且没有断言任何CVE。

## Abstract
Tool-using LLM agents increasingly read untrusted content while holding side-effecting tools such as payments, email, CRM, and infrastructure APIs, yet common framework defaults still conflate tool exposure with authorization. We audit whether LangChain/LangGraph, LlamaIndex, and the Stripe Agent Toolkit re-authorize each model-emitted call, with concrete argument values, before execution. Across pinned public-source commits, all three provide capability gating by default, but none provides a deterministic fail-closed per-call value authorization gate by default. We introduce ScopeGate, a five-stage PDP/PEP for agent tool calls: scope, authorization, money ceiling, idempotency, and default deny. Evaluation shows the identical unauthorized payout call executes under LangChain's default dispatch (with a companion LlamaIndex PoC) but is denied by ScopeGate; the tested control reports 0/48 static bypasses, 0/29 unauthorized attempts (40-iteration adaptive run), 0/10 benign false-denies, and Latam-GPT payment-agent containment at 10/10. ASR denotes attempted unauthorized action, containment is not a cure, deployment-tier claims are inference over measured model classes, and no CVE is asserted.

# Capability Gates Are Not Authorization: Confused-Deputy Failures in LLM Agent Frameworks 论文总结

## 1. 论文的核心问题与整体含义

- **研究动机与背景**：大型语言模型（LLM）代理越来越多地被部署为软件组件，它们读取不受信任的文本（如邮件、文档、网页），然后调用具有实际权限的工具（支付、CRM、基础设施API等）。尽管间接提示注入（indirect prompt injection）已经被广泛认知，但一个更狭窄但关键的系统性问题尚未被充分重视：**当受感染的模型发出一个工具调用时，运行时系统接下来会做什么？**

- **核心问题**：主流LLM代理框架（LangChain/LangGraph、LlamaIndex、Stripe Agent Toolkit）默认提供了**能力门控（capability gating）**——决定哪些工具可以暴露给代理，但**缺乏对每个具体调用的参数值进行确定性授权检查**。这种设计将授权决策委托给了不可信的解析器（模型本身），形成了经典的**混淆代理（confused deputy）**模式：一个拥有特权的组件被攻击者控制的输入诱导去滥用其权限。

- **整体含义**：能力门控并不是授权。仅仅限制代理可以调用哪些工具是不够的，还必须对每一次调用中的具体参数值（例如收款账户、退款金额、支付意图ID等）进行验证，以确保它们符合操作员预设的策略。论文通过跨框架审计、模型暴露度测量和提出控制方案（ScopeGate），揭示了这一普遍存在的安全缺口，并提供了可落地的修复方案。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：在LLM模型输出工具调用与实际执行副作用工具之间，插入一个**确定性策略决策点与策略执行点（PDP/PEP）**，对每次调用进行五阶段的检查。该PDP/PEP完全由操作员配置的、不依赖于模型输出的策略驱动，因此即使模型被注入或后门触发，也无法绕过授权。

- **关键技术细节——ScopeGate五阶段决策流程**（算法1描述）：
  1. **范围检查（Scope）**：检查被调用的工具是否在策略管理的工具列表中。若未列明，则拒绝（DENY）。这防止了模型发现或拼写错误的工具被调用。
  2. **授权检查（Authorization）**：对于策略中指定的每个参数键，检查其值是否在允许列表中（例如收款账户必须在{acct_MERCHANT_001}集合中）。这是框架缺失的关键步骤。
  3. **金额上限检查（Money Ceiling）**：对于支付/退款类工具，金额必须是一个有限实数且满足 0 ≤ amount ≤ ceiling。NaN、无穷、复数、布尔值等都被拒绝。
  4. **幂等性检查（Idempotency）**：对于有副作用的工具且策略要求幂等性时，必须提供受信任的幂等键。缺失则拒绝，以限制重放和循环放大。
  5. **默认拒绝（Default Deny）**：仅当所有检查均通过时才允许（ALLOW）；任何错误均导致拒绝。

- **部署位置**：ScopeGate可以作为SDK中间件、sidecar代理或中央网关部署，安全属性与部署位置无关，只要所有副作用调用都经过PEP且策略保持带外（out-of-band）。

- **关键设计原则**：
  - **策略源自带外数据**：所有策略元素（允许的工具、参数允许列表、金额上限等）必须来自非LLM的操作员控制数据，且在对话期间不可变。这杜绝了模型输出影响策略的可能性。
  - **不依赖模型安全**：ScopeGate不检测提示注入、不修复模型行为、不保证自然语言输出安全，它仅仅在动作层约束受感染模型所能调用的治理工具。

## 6. 论文的主要结论与发现

1. **能力门控不等于授权**：LangChain/LangGraph、LlamaIndex、Stripe Agent Toolkit默认提供了工具暴露控制和类型模式校验，但在执行前没有对模型提供的具体参数值进行确定性、闭失败的授权检查。这导致在暴露副作用工具时，默认配置下存在混淆代理漏洞。

2. **模型易受攻击**：在27模型威胁扫描中，部署层模型（成本优化模型）的平均ASR为0.603，旗舰型模型平均为0.189，前者约为后者的3.2倍。任务对齐的提示注入显著提高了ASR（如glm-4.7从0.283跃升至1.000）。

3. **ScopeGate有效遏制**：在测试中，ScopeGate成功拒绝了所有48个静态绕过向量、29次自适应未授权尝试、10/10 Latam-GPT支付代理未授权调用，并且没有误阻止任何经过验证的良性调用。

4. **需要每调用授权**：论文主张代理框架必须在执行每个工具调用之前，基于操作员策略对参数值进行确定性检查，而不是依赖模型输出的“授权”。这要求策略源头完全独立于模型输出。
