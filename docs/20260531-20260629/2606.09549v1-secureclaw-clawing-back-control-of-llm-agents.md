## 摘要
使用工具的大型语言模型（LLM）智能体面临两种不同的安全缺陷：未经授权的外部操作以及在最终输出检查干预之前运行时内部敏感明文的暴露。现有防御通常只保护一个边界，即规划器/运行时或动作接收端，因此它们本身无法同时确保两个表面的安全。我们提出了SecureClaw，一种双边界架构，在效果接收端实施授权，在读取边界实施明文隔离。敏感读取通过一个可信网关，该网关将原始值替换为不透明句柄，并在评估部署中，将受限摘要作为显式的解分类接口。改变外部状态的写入遵循PREVIEW→COMMIT协议，其中只有可信执行者才能提交策略授权的确切规范请求。运行时仍然可以基于摘要和符号引用进行规划，但不能直接解引用秘密或执行副作用。在AgentDojo、AgentLeak和Agent Security Bench（ASB）上，SecureClaw是我们在通用测试平台中评估的唯一一种防御，它同时保持了可用的任务效用，并在ASB上实现了0%的攻击成功率（ASR），在AgentDojo上为0.64%的ASR，在AgentLeak的攻击平等通道上总体泄露为3.23%，该通道衡量最终输出和内部中继泄露。

## Abstract
Tool-using large language model (LLM) agents face two distinct security failures: unauthorized external actions and exposure of sensitive plaintext inside the runtime before any final output check can intervene. Existing defenses usually protect one boundary, either the planner/runtime or the action sink, and therefore do not by themselves secure both surfaces. We present SecureClaw, a dual-boundary architecture that places authorization at the effect sink and plaintext confinement at the read boundary. Sensitive reads pass through a trusted gateway that replaces raw values with opaque handles and, in the evaluated deployment, bounded summaries as an explicit declassification interface. Writes that change external state follow a PREVIEW$\rightarrow$COMMIT protocol in which only a trusted executor may commit the exact canonical request authorized by policy. The runtime can still plan over summaries and symbolic references, but cannot directly dereference secrets or perform side effects. Across AgentDojo, AgentLeak, and Agent Security Bench (ASB), SecureClaw is the only defense we evaluate in a common harness that simultaneously retains usable task utility and achieves 0\% attack success rate (ASR) on ASB, 0.64\% ASR on AgentDojo, and 3.23\% overall leak on AgentLeak's attacked parity lane, which measures final-output and internal-relay leakage.

### SecureClaw：重新夺回对LLM智能体控制权的双边界架构

#### 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：使用工具的大型语言模型（LLM）智能体面临两种独立且致命的安全风险：
    - **未授权的外部操作**（效果面）：攻击者通过间接提示注入，控制智能体执行非预期的外部动作，如发送邮件、转账、修改数据等。
    - **运行时敏感明文泄露**（明文面）：在最终输出检查之前，敏感数据（如邮件正文、发票金额）以明文形式存在于不可信的运行时环境中，可以通过智能体间的消息、共享内存等内部通道泄露出去。
- **现有防御的不足**：现有防御通常只针对一个边界（例如，只保护动作接收端或只过滤运行时内容），无法同时处理这两种风险。一种防御的缺失会危及系统的整体安全。
- **论文含义**：要安全地部署LLM智能体，必须将“效果授权”与“明文隔离”作为两个不可相互替代的、同等重要的安全支柱来处理。SecureClaw通过架构层面的双边界设计，从系统根基上解决这个问题。

#### 2. 方法论：核心思想与关键技术细节

- **核心思想**：不可信的智能体运行时不应控制外部效果何时发生，也不应控制原始敏感数据存在于何处。安全边界应设置在可信与不可信组件之间。
- **架构**：SecureClaw将安全执行与任务规划分离，主体架构包括：
    1.  **不可信运行时**：负责规划、调用工具，但不再被信任用于授权和机密性。
    2.  **可信网关**：所有敏感读取必须经过网关。
    3.  **可信句柄存储**：安全存储敏感数据的原始明文。
    4.  **可信策略引擎**：对规范化的请求进行授权或拒绝。
    5.  **可信执行器**：唯一能向外部效果汇（如发送邮件API）提交操作的组件。
- **关键技术细节**：
    - **读路径（明文面防护）**：网关将敏感数据存储到可信句柄存储中，并向运行时返回一个**不透明句柄**（一个高熵的符号引用）和一个**受限摘要**（明确限定的、去分类化的摘要）。运行时只能操作句柄和摘要，无法直接访问原始明文。
        - 摘要被明确定义为“显式的解分类接口”，是有意为之的、可审计的信息释放通道。
    - **写路径（效果面防护）**: 采用**PREVIEW → COMMIT协议**。
        - **预览**：运行时只能`提议`一个效果动作（如“发送邮件”），并提供 `intent`、`inputs`等。可信网关会补充 `caller`、`session`等鉴权信息，并对整个请求进行规范化（Canonicalization）和哈希绑定，生成唯一摘要，交由策略引擎授权。
        - **提交**：可信执行器独立重新计算请求的哈希摘要，并验证策略引擎返回的授权凭证。只有凭证有效且与当前请求完全匹配时，执行器才会真正提交外部效果。任何篡改（如修改接收者）都会导致校验失败。
    - **拒绝感知恢复**：当`提交`被拒绝后，系统不会终止，而是提供一个安全的延续路径，如“草拟待确认”，这在不牺牲安全性的前提下恢复了大部分任务效用。
- **公式/算法流程**（文字描述）：协议1 `PREVIEW → COMMIT` 流程如下：
    1. 运行时提议动作。网关补充鉴权上下文，形成完整请求 `ρ`。
    2. 网关规范化 `ρ` 并计算绑定摘要 `h = HMAC-SHA256(k_bind, Canon(ρ))`。
    3. 网关向策略引擎查询，获得绑定到 `h` 的授权凭证。
    4. 执行器独立重新计算 `h`，验证授权凭证，并检查新鲜度、重放状态、确认令牌等。
    5. 所有检查通过，执行器执行效果。否则，执行器失败关闭。

- **核心结论**：保护LLM智能体必须分离“效果控制”与“明文访问”两个边界，并且两者不可相互替代。

- **问题解耦清晰**：首次明确区分了“效果面”和“明文面”两个不可相互替代的安全空间，并针对性地设计了不同机制。

- **不保证语义正确性**：SecureClaw不能保证策略允许的动作是用户期望的。策略本身可能过于粗粒度，导致合法但错误的动作被执行。
