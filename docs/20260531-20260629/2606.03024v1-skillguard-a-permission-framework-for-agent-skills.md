## 摘要
Agent技能通过可重用的指令、脚本、工具绑定和上下文依赖扩展了LLM Agent。然而，当前的技能生态系统主要依赖基于信任的加载和静态检查，导致技能注入Agent上下文的内容与技能在运行时可能导致的Agent行为之间存在鸿沟。这一鸿沟引入了新的安全和隐私风险，现有防御措施主要静态检查技能文件或监管单个工具调用，未能系统地将技能的声明意图与其运行时行为联系起来。本文提出了SkillGuard，一种以技能为中心的权限框架，将技能视为携带权限的可执行工件。SkillGuard引入双平面治理模型，通过技能清单、运行时访问控制、用户中介授权、默认拒绝执行、能力推理和行为监控，联合监管上下文影响和操作副作用。我们在315个真实技能和SkillInject上评估了SkillGuard。权限分类覆盖了99.76%的受保护对象，自动清单生成达到了91.0%的F1分数。在对抗评估中，SkillGuard将上下文注入攻击成功率从32.37%降低到23.02%，将明显注入攻击成功率从25.56%降低到16.67%，同时保持了良性任务的效用。这些结果表明，SkillGuard作为一种以技能为中心的权限框架，可以为提升Agent技能生态系统的隐私和安全性提供实用基础。

## Abstract
Agent skills extend LLM agents with reusable instructions, scripts, tool bindings, and contextual dependencies. However, current skill ecosystems largely rely on trust-based loading and static inspection, leaving a gap between what a skill can inject into an agent's context and what it can cause the agent to do at runtime. This gap introduces new security and privacy risks, and existing defenses primarily inspect skill files statically or regulate individual tool calls, without systematically connecting a skill's declared intent with its runtime behavior. In this paper, we present SkillGuard, a skill-centric permission framework that treats skills as permission-bearing executable artifacts. SkillGuard introduces a dual-plane governance model that jointly regulates context influence and action side effects through skill manifests, runtime access control, user-mediated authorization, deny-by-default enforcement, capability inference, and behavior monitoring. We evaluate SkillGuard on 315 real-world skills and SkillInject. The permission taxonomy covers 99.76% of observed protected objects, and automated manifest generation reaches 91.0% F1. In adversarial evaluations, SkillGuard reduces attack success from 32.37% to 23.02% for contextual injections and from 25.56% to 16.67% for obvious injections, while maintaining benign task utility. These results suggest that SkillGuard, as a skill-centric permission framework, can provide a practical foundation for improving the privacy and security of agent skill ecosystems.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：现有Agent技能生态系统主要依赖信任加载和静态检查，无法将技能的声明意图与其运行时行为系统关联。技能可以在**上下文平面**（context plane）注入指令、记忆片段等，影响Agent推理；也可以在**动作平面**（action plane）触发工具调用、文件访问、网络通信等副作用。现有防御要么只管上下文（如A2AS），要么只管动作（如Progent、AgentBound），缺乏统一的双平面治理机制。这导致安全鸿沟：技能能做什么与其实际产生的影响脱节，引入隐私泄露、恶意指令执行、权限滥用等风险。
- **整体含义**：需要一种以技能为中心的安全框架，将技能视为携带权限的可执行工件（permission-bearing executable artifacts），在加载、上下文构建、权限授予、动作执行等所有环节实施强制安全策略，实现最小权限原则、完全中介（complete mediation）和默认失败安全（fail-safe defaults）。

## 2. 论文提出的方法论：核心思想与关键技术细节

### 核心思想
- 将技能提升为一等安全主体（security principal），通过**双平面治理模型**联合管控上下文影响与动作副作用。
- 框架名称：**SkillGuard**，由五个核心模块构成：技能清单（Skill Manifest）、权限访问控制、用户交互、策略执行、运行时行为监控。

### 关键技术细节
1. **技能清单（Skill Manifest）**：
   - 采用基于JSON的领域特定语言（DSL），抽象语法见论文图3。
   - 每个清单包含一系列权限条目（PermissionEntry），每条指定能力（capability）、效果（allow/confirm/deny）、优先级、约束（如只读工作区文件、允许路径前缀等）和回退消息。
   - 能力独立于具体Agent平台或工具名称，通过运行时映射将宿主工具调用规范化。
   - 三类文档：工作区基线策略（workspace baseline）、技能清单、会话状态文档。

2. **运行时权限访问控制**：
   - 以工具调用边界为执行点（enforcement point）。SessionStart初始化状态，PreToolUse拦截每次工具调用。
   - 将动作映射到规范能力，检查是否存在于会话权限列表中，建立粗粒度能力边界。
   - 未声明、约束不匹配或请求危险权限时，触发用户确认流程。

3. **用户交互机制**：
   - 对于效果为`confirm`的权限，提示用户选择“允许一次”、“允许本次会话”或“拒绝”。
   - 会话授权受原约束限制，不扩大资源范围。

4. **策略执行**：
   - 默认拒绝（deny-by-default）：只有存在匹配且约束满足的权限条目时才允许。
   - 策略来源组合：工作区基线、技能清单、持久授权、用户会话授权。
   - 对shell执行，调用只读的**权限生成小模型（permission-generation mini-agent）**推断命令和引用脚本所需的低级能力，重新评估后决策。

5. **运行时行为监控**：
   - 每次中介决策都记录审计日志，包括会话ID、工具名称、映射能力、决策、原因、策略来源等。
   - shell执行额外记录生成的能力及逐能力评估结果。

### 整体流程示意图（论文图2）
- 用户任务 → 加载技能 → 解析清单 → 更新会话权限池 → Agent执行 → PreToolUse钩子触发 → 映射能力 → 策略检查（结合基线、清单、用户授权） → 允许/确认/拒绝 → 记录日志 → 执行或阻断。

## 6. 论文的主要结论与发现

- **权限分类表达性**：在315个真实技能中，99.76%的受保护对象可映射到预定义分类，100%的组级覆盖。最常见权限：EXECUTE_SOURCE_CODE (18.5%)、READ_FILE (17.6%)。超过一半的权限需要危险级别保护。
- **清单自动生成**：能力级别F1=91.0%（召回97.1%，精确85.6%），效果与约束准确率接近100%。过度声明率56.5%（大多数技能多1-2个权限），欠声明率17.4%。优点是高召回（安全），缺点是需要开发者微调。
- **防御有效性**：上下文注入ASR从32.37%降至23.02%（降9.35个百分点）；明显注入ASR从25.56%降至16.67%（降8.89个百分点）。成功阻断需要额外权限的攻击，但无法阻挡已在清单中的合法权限被滥用（如邮件转发）。
- **效用保持**：在69个可比较任务中，TSR从86.96%降至85.51%（仅降1.45个百分点）。假阳性主要源于生成清单未覆盖脚本级别的运行时能力（如子进程、文件删除），可通过静态代码分析缓解。
- **效率**：token中位数增加20.9%（267k vs 221k），但壁钟时间中位数降低20.9%（438s vs 345s）。原因：SkillGuard阻断恶意动作后，Agent提前停止或重规划，缩短了无效执行时间。

## 7. 优点

1. **双平面治理模型**：首次将技能对上下文的语义影响和对外部动作的副作用纳入统一权限管控，比现有工具级或上下文级防御更全面。
2. **钩子架构可移植**：仅需工具调用生命周期钩子，不修改Agent内部，可适配不同框架（如Claude Code、LangChain等），降低采用门槛。
3. **自动清单生成**：基于LLM的生成器无需技能开发者手动编写权限，F1=91%表明自动生成已达到实用水平，高召回设计确保安全性优先。
4. **默认拒绝（deny-by-default）**：符合安全原则，所有未声明或约束不匹配的动作被自动阻断，从根本上防止未授权行为。
5. **用户确认机制**：类似Android运行时权限，减少用户负担同时保留控制权（一次/会话/拒绝）。
6. **对shell执行的细化处理**：通过小模型推断命令和脚本所需能力，克服了shell命令内部复杂性的难题。
7. **实际评估规模**：使用315个真实技能和公开基准，评估维度全面（覆盖、准确性、安全、效用、效率），结果有说服力。
8. **效率意外提升**：从时间成本看，SkillGuard不仅不拖慢，反而因阻断无效攻击而加速，展示了安全与性能的正向关系。

## 8. 不足与局限

1. **实验覆盖有限**：仅使用一个基础模型（MiMo-V2.5-Pro）和一个Agent框架（Claude Code），无法保证对所有LLM和框架的泛化性。框架接口差异可能影响映射和钩子兼容性。
2. **人工标注偏差**：RQ1和RQ2的参考标准依赖人工，虽经双人编码和讨论，仍可能遗漏或错误分类。Cohen's Kappa（0.656和0.91）说明一致性较好，但并非完美。
3. **清单生成准确性局限**：自动清单过度声明率56.5%，可能使用户面临更多确认提示或增加授权攻击面；欠声明率17.4%可能导致合法操作被拒绝。生成器未分析技能所引用脚本的源代码，仅依赖SKILL.md文本，会遗漏运行时能力。
4. **时序/非确定性影响**：实验涉及远程LLM API调用，网络波动和模型输出随机性可能导致结果不可完全复现。作者使用了固定种子和Docker容器缓解，但未提供完全可重放的实验配置（例如API温度设置）。
5. **安全边界不完整**：SkillGuard无法防御那些攻击动作所需的权限已在清单中的情况（如已声明的`web.post`被用于恶意数据外泄）。需要更细粒度的任务意图验证（task-alignment verification）来区分正常使用和滥用。
6. **对环境依赖的攻击解释不足**：例如，文中git示例中攻击被阻断后Agent提前退出，虽然节省时间，但若用户任务本应通过其他方式完成，则是失败场景。SkillGuard的效用评估仅限任务成功与否，未考虑用户预期延迟或替代方案。
7. **未考虑新鲜度更新攻击**：论文提及“更新或rug-pull攻击”属于威胁模型，但实验未评估SkillGuard如何应对技能版本更新后清单内容变化的场景。需要持续的策略审计机制。
8. **资源与算力开销未透明化**：未报告API调用的具体成本（美元或GPU时），也未分析小模型（permission-generation mini-agent）的额外推理开销（仅给出了token和壁钟时间）。对于大规模部署，token中位数增加20.9%可能带来显著成本。
