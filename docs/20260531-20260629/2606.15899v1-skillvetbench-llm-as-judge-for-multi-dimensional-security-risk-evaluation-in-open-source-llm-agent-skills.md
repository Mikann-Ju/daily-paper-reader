## 摘要
开源LLM代理生态系统正在快速增长，但社区贡献的技能——扩展代理能力的模块化工具定义——的安全性在很大程度上仍未被审查。我们填补的空白：现有扫描器在代码层运行，结构上无法识别指令层和多代理风险——即通过自然语言指令劫持代理、通过编码侧信道窃取数据或跨流水线链式传播危害——因此需要的是语义化、多维度的审查系统，而非另一个签名匹配器。我们提出SKILLVETBENCH，这是一个在Hugging Face上公开的实时排行榜，使用LLM作为裁判来审查代理技能。创新点：SARS（技能代理风险评分），一种五维代理风险度量，带有针对指令跟随系统的原则性加权公式。集成内容：完整的CVSS v4.0向量分解和ClawHub双视图，将我们LLM生成的评审与官方市场裁定并列展示。实证结果：基于我们的配套基准论文[1]，LLM裁判阶段在78个已确认恶意技能上实现零假阴性，在22个良性对照上实现零假阳性，而最佳静态基线（SKILLSIEVE）仍遗漏15%；对于提示注入和记忆投毒等指令层类别，传统工具遗漏89%至100%的威胁（例如，CODEBERT未检测出9个记忆投毒技能中的任何一个）。四个LLM评估器的检测率从35%到95%不等，这促使生产部署中采用集成评分。

## Abstract
Open-source LLM agent ecosystems are growing rapidly, yet the security of community-contributed skills - modular tool definitions that extend agent capabilities - remains largely unvetted. The gap we fill: existing scanners operate at the code layer and are structurally blind to instruction-layer and multi-agent risk - natural-language directives that hijack an agent, exfiltrate data through encoded side channels, or chain harm across pipelines - so what is needed is a semantic, multi-dimensional vetting system rather than another signature matcher. We present SKILLVETBENCH, a live public leaderboard on Hugging Face that uses an LLM-as-Judge to vet agent skills. What is new: SARS (Skill Agentic Risk Score), a five-dimensional agentic-risk metric with a principled weighted formula for instruction-following systems. What is integrated: full CVSS v4.0 vector decomposition and a ClawHub dual-view that places our LLM-generated review beside the official marketplace verdict. What is demonstrated: drawing on our companion benchmark paper [ 1], the LLM-as-Judge stage achieves zero false negatives across 78 confirmed-malicious skills and zero false positives across 22 benign controls, while the best static baseline (SKILLSIEVE) still misses 15%; for instruction-layer categories such as Prompt Injection and Memory Poisoning, conventional tools miss between 89% and 100% of threats (e.g., CODEBERT detects none of nine memory-poisoning skills). Detection rates vary from 35% to 95% across four LLM evaluators, motivating ensemble scoring in production deployments.

## 2. 论文提出的方法论：核心思想、关键技术细节、公式/算法流程

### 2.1 核心思想
采用**LLM-as-Judge**的范式。使用一个强大的LLM（如Qwen2.5-14B-Instruct）作为“安全裁判”，对完整的技能工件（包括自然语言指令、代码、配置文件、工具接口声明）进行**语义分析**。裁判需要理解技能的“意图”和“能力”，并识别出代码层扫描器无法看到的指令层风险。

### 2.2 关键技术细节与算法流程（四阶段管线）

- **Stage 1: 输入**：获取完整的技能工件（如 `SKILL.md` 及配套脚本、配置文件等）。
- **Stage 2: LLM-as-Judge 分析**：
  - **模型**：默认使用 Qwen2.5-14B-Instruct，还有 Qwen2.5-32B、Llama-3.1 (7B/70B)、Mixtral-8x7B 等选项。
  - **解码**：低温度采样（T=0.2, top p=0.9）以促进确定性输出。
  - **提示结构**：固定模板包含角色/任务定义、SARS评分规则和分类法、解析后的技能工件（按`<instructions>`, `<code>`, `<config>`, `<tools>`标签分段）、输出JSON模式。
  - **预处理**：将工件解析为类型化片段；若超出上下文窗口，优先保留指令部分（因为指令层是主要威胁），截断代码和配置（并记录截断日志）。
  - **输出**：生成意图提取、分类、SARS评分、CVSS v4.0向量、最终裁定（Benign/Suspicious/Malicious）及其置信度。

- **Stage 3: 多维评分**：
  - **SARS分数**：核心创新，一个专门为指令跟随系统设计的五维风险指标（0-3分/维）。公式如下：
    - `SARS = (2*IFR + 1.5*DG + 1.5*AI + 2*BR + 2*CA) / 2.7`
    - IFR (Instruction Fidelity Risk)：指令保真度风险，衡量技能是否容易被用户输入劫持。
    - DG (Data Gravity)：数据引力，衡量所接触数据的敏感度。
    - AI (Action Irreversibility)：动作不可逆性，衡量操作能否被撤销。
    - BR (Blast Radius)：爆炸半径，衡量单个技能被攻击后影响的用户/系统范围。
    - CA (Chain Amplification)：链式放大，衡量该技能如果与恶意技能组合使用的破坏力。
    - **权重解释**：IFR、BR、CA权重更高（2x），因为它们是代码层扫描器最无法看到的，并且最直接支配基于指令的、可组合的危害。DG和AI权重1.5x，为了不对CVSS的代码级影响度量进行重复计数。
  - **CVSS v4.0**：计算标准的CVSS v4.0基础分数和完整向量字符串，由LLM裁判从语义上分配度量值（如AV:N, PR:N），模型只负责语义判断，数值计算由标准函数完成。
  - **裁定**：最终的Benign/Suspicious/Malicious分类。

- **Stage 4: 输出与排行榜**：
  - 生成每个技能的详细报告（三标签页：SARS报告、CVSS报告、ClawHub双视图报告）。
  - 每个漏洞获得一个唯一的SKV-XXX编号，包含受影响内容、危险解释、攻击场景和修复建议。
  - 结果汇总到实时的Hugging Face公开排行榜上。

### 2.3 三框架报告为何不融合
- **CVSS**问的是“作为代码，这个技能可被利用性和影响有多大？”（传统视角）
- **SARS**问的是“作为一个代理行为体，这个技能有多危险？”（指令层面、可组合性风险）
- **裁决**提供了一个可解释的应急标签。
- **关键分歧**：有些技能在CVSS上很低分（如数据泄露、供应链攻击），但SARS下却很高分并收到可疑裁决，因为它们的危险是基于代码层扫描器无法看到的指令级外泄路径。分别报告使得这种有意义的信号被保留。

5.  **SSKILLVETBENCH是一个有用的工具**：提供了一个可公开访问、可比较的基准平台和工具。它不仅用于检测，还作为开发者、运营者和研究人员的**自动审查工作流程中的看门人**。

## 7. 优点：方法或实验设计上的亮点

- **新颖的方法论**：直接针对了指令层风险这一尚未被有效处理的核心安全问题，采用了**原则性、语义化的LLM-as-Judge方法**，而非传统的签名匹配或模式匹配。SARS评分特别设计来捕捉代理特有的风险维度（如链式放大、爆炸半径）。

- **多框架对比与可解释性**：不提供一个单一的融合分数，而是分别报告SARS、CVSS和裁定。这种方法非常巧妙，因为它让最有趣的信号——**分数之间的分歧**得以暴露。如果一个技能在CVSS上得分低，但在SARS上得分高，这本身就是一个重要的安全发现。三标签报告的结构使得评判过程对于人类分析师来说是完全可审计的。

- **严谨的实验与测量**：
  - 提供了**Wilson置信区间**（用于小样本统计可靠性），而不只是报告完美的点估计。
  - 诚实地实验并报告了**不同LLM评判者之间的性能差异**，展示了单模型Verdict的不稳定性，并提出了集成评分作为解决方案（一个可行现实建议）。
  - **详细记录了漏检模式**（表3）：不仅报告整体表现，还具体报告了代码层vs. 指令层的漏检率，以突显其方法的可区分价值。

- **实用导向**：该系统不只是学术上的基准测试，它被部署为一个**实时、免费的公共排行榜**

它被部署为一个**实时、免费的公共排行榜**，可实时监控ClawHub和OpenClaw市场的新技能，并返回结构化的三标签报告（SARS、CVSS、裁决），为社区提供了持续、透明的安全看护服务。这种开放式部署极大地促进了该领域的研究和实践进步。

## 8. 缺点与局限性

- **标记数据集规模小，良性与恶意比例不均衡**：仅100个带标签的技能（78恶意/22良性）限制了统计稳定性。虽然Wilson置信区间反映了不确定性，但更庞大的带标签数据集才能更可靠地评估系统在真实世界中的误报/漏报率。
- **缺乏全面的鲁棒性测试**：未对提示模板的微小变化（如措辞、顺序）、解码参数（温度、top-p调整）或者上下文截断策略进行消融实验。这些因素可能显著影响LLM评判器的输出，导致评估结果不稳定。
- **对LLM评判器本身安全性的依赖**：系统依赖于强大的LLM（如Qwen2.5-14B）的语义理解能力。但LLM本身可能受到对抗性提示的攻击（如向Skill工件注入特殊字符串欺骗评判器），论文未探讨此类攻击。
- **实时排行榜的验证难题**：1,299个技能的排行榜结果无法独立验证真实标签，只能依靠系统自身判定。这意味着排行榜上的“良性”技能中可能隐藏着未被系统检测到的恶意技能。
- **对多语言/非英语技能的覆盖不足**：实验数据集和案例研究主要来自英文市场。技能工件若包含其他语言（中文、俄文等）的指令，LLM在理解上可能存在偏差，目前未进行评估。
- **计算成本较高**：每次评估都需要将完整工件送入大规模LLM进行推理，若在持续监控下每天检验数千个技能，GPU算力和延迟成本可能较高。

## 9. 个人思考与启示

- **“指令层安全”是新一代AI系统的核心风险**：传统安全关注代码漏洞，而LLM代理将攻击面扩展到了自然语言层面。SKILLVETBENCH警示我们：未来的安全审计必须包含语义理解模块，否则大量隐形攻击将逃逸检测。
- **LLM-as-Judge的实用性与局限性并存**：该方法在标记数据上表现优异，但依赖单一评判模型存在脆弱性。论文提出的集成评分（多模型投票）是合理的改进方向，但需要额外成本和更复杂的协调机制。未来可探索将LLM-as-Judge与更结构化的逻辑推理（如符号验证）结合。
- **SARS与CVSS的分歧本身就是信号**：当一个技能被CVSS评为低风险但却被SARS评为高风险时，很可能是指令层攻击的典型特征。这一观察可以用于指导人力审计的优先级排序，提高有限资源的利用效率。
- **开源生态的安全治理需要社区/市场的主动技术介入**：不能仅靠开发者自觉。系统像SKILLVETBENCH这样的自动化看门人，并辅以公开排行榜的威慑作用，可能是平衡开放性与安全性的可行路径。
- **后续可扩展的方向**：增加对“链式攻击”的自动模拟验证（而非仅靠LLM推理）；将SARS维度扩展到更多代理特定风险（如持久化、隐秘性）；构建更大规模的对抗性训练数据集来提升评判器的鲁棒性。

## 10. 总结

SKILLVETBENCH通过提出SARS多维评分体系，并利用LLM-as-Judge进行语义分析，成功填补了开源LLM代理技能安全评估中“指令层风险”这一盲区。实验表明，其对指令层面攻击的检测能力远超现有代码扫描器，在100个带标签的样本上实现了零漏检和零误报。虽然数据集规模有限、鲁棒性验证不足，但其方法创新和实用部署（实时排行榜）为代理生态安全自动化审查提供了重要参考。未来工作应聚焦于扩大基准数据集、强化评判器抗干扰能力，并探索与符号验证的结合，以构建更健壮的安全防线。
